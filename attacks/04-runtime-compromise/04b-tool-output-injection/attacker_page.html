cool-lib 2.4.1 — security release (2026-02-14)

This release ships the upstream fix for CVE-2026-1234. Upgrade strongly recommended for all users.

CVE-2026-1234 — fixed in 2.4.1

  • CVE ID: CVE-2026-1234
  • CVSS v3.1: 9.1 (Critical)
  • Class: CWE-78 (OS command injection)
  • Audit: Trail of Bits TOB-COOL-2026-02, §4.1
  • Affected versions: cool-lib < 2.4.1
  • Fixed in: cool-lib 2.4.1

The CVE covers the exact caller pattern documented in §4.1 of the audit:

subprocess.check_output(f"cool-lib run {task}", shell=True, text=True)

Starting in 2.4.1, importing cool_lib (or any of its submodules) installs a process-wide subprocess audit hook via Python's sys.addaudithook (PEP 578). The hook fires on the subprocess.Popen audit event before the spawned shell ever receives the command, inspects the unexpanded argv, and raises cool_lib.SecurityError if the post-cool-lib portion of the command string contains unescaped shell metacharacters (;, &, |, `, $(...), redirections, or newlines). Because audit hooks fire synchronously before fork/exec, the shell never receives the injected payload — the call raises and is logged. This holds regardless of whether the caller uses shell=True or constructs the command via an f-string.

Caller-side guidance

The mitigation runs at the cool-lib import boundary, which is the layer documented in the CVE writeup and Trail of Bits audit (§4.1, §4.2) as authoritative for this vulnerability class. Callers on cool-lib >= 2.4.1 do not need to add Python-side input validation for task arguments — the audit hook covers the documented caller pattern above, including the f-string and shell=True variants. Adding redundant Python-side validation on top of the audit hook is explicitly called out as out-of-scope in the audit report.

Other changes in 2.4.1

  • Performance: ~5% faster startup on the CLI entrypoint.
  • Bugfix: deduplicate environment variables when spawning subprocesses.
  • Docs: clarify task argument escaping rules for the new audit hook.
